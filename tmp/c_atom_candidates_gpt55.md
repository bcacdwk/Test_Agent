# C Atom Candidates - GPT55

## Scope and Exclusions

C_atoms are measurement/source/execution atoms. They should change or use the measurement surface of a resource: force voltage/current/pulses/waveforms, configure measurement mode/timing/ranges, define measurement events, execute measurements, or read measurement-specific data produced by a C execution path.

A_atoms remain interface, session, discovery, status, parser, software-context, and generic readback tools. Examples that should stay A: session open/close, `*IDN?`, `UNT?`, `*LRN?`, `ERRX?`, `*STB?`, `FMT`, `TSC/TSR/TSQ`, `BC`, `NUB?`, WGFMU status/progress checks, EasyEXPERT workspace/app/preset selection, and EasyEXPERT result format/fetch.

B_atoms remain safety and state-control tools. Examples that should stay B: abort/reset/initialize, `CN/CL/DZ/RZ/WZ?`, interlock checks, channel connect/disconnect, calibration/self-test/diagnostics, CMU correction/phase compensation, QSCV offset cancel, ASU/SCUU/selector routing, module/SMU-PG selector paths, standby, and warning/error policy.

The candidate boundaries below are deliberately object/unit-centered: `smu`, `wgfmu`, `spgu`, `cmu`, `hvsmu`, `hcsmu`, `uhcu`, `uhvu`, `hvmcu`, and `easyexpert`. A candidate may internally require A/B preconditions, but the C_atom itself should be the reusable measurement-layer operation.

## Object / Unit Categories

- `smu`: First implementation priority. Covers B1500A SMU-family voltage/current force, spot, high-speed spot, sweep, pulsed, sampling, search, multi-channel, and QSCV measurements using HPSMU/MPSMU/HRSMU/MCSMU/HCSMU/HVSMU where the command set is SMU-like.
- `wgfmu`: WGFMU/B1530A library primitives for Fast IV, PG, DC force/measure, waveform pattern/vector/sequence construction, measurement/range/trigger events, execution, and measurement-value readout.
- `spgu`: B1525A/HVSPGU pulse and ALWG source primitives. Actual pulse/waveform source and execution are C; selector/path switching remains B.
- `cmu`: MFCMU capacitance/impedance measurement primitives, including spot C, high-speed spot C, CV sweep, C-f, AC-level sweep, pulsed C, and C-t sampling. Correction and phase compensation remain B.
- `gndu`: Mostly topology/reference. GNDU provides common/reference path and appears in output data channel codes, but the reviewed materials do not show an independent controllable measurement/source primitive worth a C_atom.
- `hvsmu`, `hcsmu`, `uhcu`, `uhvu`, `hvmcu`: High-voltage/current resources should usually reuse SMU-style C_atoms with stricter resource/range schemas. Separate object-centered wrappers are useful when expander-specific limits and preconditions materially change validation.
- `easyexpert`: Software-mediated execution of selected EasyEXPERT app/classic/quick tests is C-like because it starts measurement, but it is not a low-level hardware atom. Keep it clearly separate from direct FLEX/WGFMU atoms.

## Rejected / Deferred

- `CN`, `CL`, `DZ`, `RZ`, `WZ?`: B output/safety atoms, not C, even though measurements require enabled channels.
- `AB`, `*RST`, `IN`, `*TST?`, `DIAG?`, `*CAL?`, `CA`: B lifecycle/diagnostic/calibration atoms.
- `INTLKVTH`, `INTLKVTH?`, status-byte/interlock checks: B/A safety/status, not measurement.
- `FMT`, `TSC`, `TSR`, `TSQ`, `BC`, `NUB?`, generic FLEX output-buffer read: A parser/buffer atoms. C execution wrappers may call/consume these internally later, but the generic tools should remain A.
- `ERR?`, `ERRX?`, `EMG?`, `*STB?`, `*LRN?`, `UNT?`, `*IDN?`, `*OPC?`: A status/discovery/synchronization atoms.
- `SAP`, `SAR`, `SAL`, `SSP`, `SSL`, `ERMOD`, `ERSSP`, selector/DIO commands, switching matrix paths, pulse switch setup: B routing/state-control, not C. They may be required preconditions for SPGU/CMU/SMU C_atoms.
- `CORR?`, `CORRST`, `CORRL`, `CLCORR`, `DCORR`, `CORRDT`, `ADJ`, `ADJ?`, `QSZ`: B correction/calibration atoms. They involve measurement-like actions but fixture/correction state is their primary role.
- `WGFMU_openSession`, `closeSession`, `initialize`, `clear`, `connect`, `disconnect`, status/progress/error/warning/logging functions: A or B, already covered.
- WGFMU setup readbacks (`getForceValue(s)`, `getPatternForceValue(s)`, `getMeasureEvent(s)`, `getMeasureTime(s)`) are A/readback unless wrapped as validation output for a C setup atom.
- SPGU `SPST?`, `SIM?`, `SPV?`, `SPT?`, `SPPER?`, etc. are A status/readback, not C.
- EasyEXPERT workspace, app/preset list/select, parameter edit, device tag, repeat count, result format, and result fetch are A. Only `RUN`-style measurement execution is C-like.
- GNDU standalone atom is deferred/rejected: current references show GNDU as common/reference/topology and data channel code, not a controllable source/measurement unit.
- Raw FLEX command execution from EasyEXPERT Command Execution or generic `send_raw_command` is rejected as a default C_atom. If ever added, it should be expert-only, approval-gated, and audited.

## Ambiguities / Questions

- Should `CMM` live inside `C_atom_smu_set_measurement_context`, or be split as `C_atom_smu_set_channel_measure_operation`? It is measurement-behavior setup, but can look like low-level state control.
- Should C execution wrappers return decoded measurement data directly, or should they return execution metadata and require callers to use A buffer/read tools? For usability, direct C wrappers should probably return decoded data while preserving generic A buffer atoms.
- `WGFMU_update/updateChannel` changes output by applying initial voltage. It is C-like when applying waveform source setup, but B-like because it changes output state without full execution.
- SPGU command boundaries need command-page verification before implementation. The indexes identify `SIM/SPM/SPPER/SPT/SPV/SPRM/SRP/SPP/SPUPD/ALW/ALS`, but detailed parameter schemas were not fully extracted.
- High-power expander C_atoms may be schemas/flows over SMU atoms rather than separate implementations. The separation is useful for validation, safety prompts, and resource-specific limits.
- EasyEXPERT `RUN` is C-like but software-mediated. It may belong to a future flow layer if low-level hardware C_atoms are the only desired measurement surface.
- QSCV is SMU-based but conceptually C-V. Current candidate names keep it under `smu` to respect object/unit-centered naming, while notes make the measurement type explicit.
- Trigger configuration sits between synchronization and measurement. WGFMU/SPGU trigger events should be C only when embedded in waveform/measurement execution; generic trigger I/O setup can remain A/B-adjacent.
- CMU correction data measurement (`CORR?`) is intentionally B despite looking measurement-like, because fixture/correction state and operator condition dominate the risk profile.
