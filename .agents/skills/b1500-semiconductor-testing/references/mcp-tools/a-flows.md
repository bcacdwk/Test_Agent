# A Flow MCP Tools

A flows are fake, orchestration-only tools that compose current `A_atom_*` tools into ordered single-transport operations. They never call `B_atom_*` or future `C_atom_*` tools.

```text
A_flow_<operation>_<interface>_<subject>
```

## A Flow Tools

`A_flow_*` tools are **fake and orchestration-only**. Each flow composes only current `A_atom_*` tools into one ordered, single-transport operation; it never calls `B_atom_*` or future `C_atom_*` tools. Like the atoms, flows return `fake: true` / `hardware_touched: false` and must not be treated as hardware control. They encode the **correct atom scheduling, side-effect flags, and warnings** so that real driver code (added later) can reuse the same sequencing and gating.

Naming pattern: `A_flow_{operation}_{interface}_{subject}` where `operation ∈ {discover, snapshot, drain, collect, prepare, select, teardown}` and `interface ∈ {flex, wgfmu, easyexpert}`.

**Boundaries (what an A_flow must never do):** no `B_atom_*`/`C_atom_*`; no reset/initialize/abort/self-test/diagnostics/self-calibration/zero-cancel; no channel enable/disable, output zeroing, routing, or correction; no measurement execution (EasyEXPERT selection is software-context only); no safety claims (a teardown is software/communication cleanup, **never** a safe shutdown — that needs a B/AB flow). The only `B_atom_*` references inside flows live in **warning text**, never as calls.

**Execution model:** atoms run **serially** on one FLEX, WGFMU, or EasyEXPERT session (the B1500A one-response query buffer and EasyEXPERT's single remote connection make concurrent transport unsafe). There is no async/parallel execution; parallelism is only legitimate across independent sessions at the application layer. Data-dependent loops (`drain_*`, catalog fan-out) are bounded by `max_errors` / `max_groups` with a `truncated` flag.

**Response envelope** (every flow): `flow`, `flow_class: "A"`, `category`, `interface`, `subject`, `fake`, `hardware_touched`, `ok`, `partial`, `destructive`, `consumptive`, `atoms_called` (ordered names actually invoked, including loop duplicates), `atom_results` (per-atom audit records preserving each atom's returned data; intentionally skipped atoms are recorded with `status: "skipped"` but excluded from `atoms_called`), `warnings`, `purpose`, plus `inputs` and a few normalized summary fields. Flag semantics: `destructive` = opens/closes a session, changes software context, or clears a buffer; `consumptive` = drains/consumes a queue or output buffer. Gated destructive steps (e.g. `allow_clear_buffer`, `force_clear`) and skipped optional reads keep the flags honest in fake mode so client gating code paths are exercised.

> Deferred: `A_flow_collect_wgfmu_progress` (WGFMU per-channel/event progress harvest) is **not implemented** — it is only meaningful during an active C-class measurement, which does not exist yet.
