# B Flow Candidates - GPT

## Scope

`B_flow_*` tools should compose only existing `B_atom_*` tools into reusable safety, reset, initialization, calibration, diagnostics, channel/output-state, path-switching, zero/correction, and state-control workflows. They may abort operations, force or disable outputs, confirm zero-output state through B-scope safety readback atoms, run self-test/calibration/diagnostics, set calibration policies, prepare ASU/SCUU paths, perform CMU correction or phase-compensation actions, run QSCV offset cancellation, initialize or self-check WGFMU state, and control EasyEXPERT standby/abort/zero-cancel state.

`B_flow_*` tools must not call `A_atom_*` or future `C_atom_*` tools. They must not open or close sessions, identify instruments or remote hosts, list modules/workspaces/catalogs, read generic status/error queues, read output/result buffers, select or execute measurement recipes, or claim measurement readiness based on communication state. If a workflow needs connection discovery, status/error readback, operation-complete waits, module inventories, result retrieval, or session teardown context, it should be an `AB_flow_*` candidate later, not a pure `B_flow_*`.

Pure B flows can use B-scope safety readbacks that already exist as `B_atom_*` tools, such as zero-output confirmation and interlock checks. They still cannot prove transport availability or full instrument health without later A-flow composition.

## Rejected / Too Broad B_flow Candidates

- `B_flow_connect_reset_and_verify_b1500`: requires A connection and identity/status queries; consider later as `AB_flow_connect_reset_verify_b1500`.
- `B_flow_safe_disconnect_b1500`: needs A session-close behavior plus B output cleanup; defer to AB flow.
- `B_flow_abort_zero_and_drain_errors`: abort/zero are B, but draining instrument errors requires A atoms.
- `B_flow_reset_initialize_and_snapshot`: reset/init are B, but snapshotting status/settings/modules is A.
- `B_flow_prepare_measurement_recipe`: any source/sweep/timing/result-buffer setup for a concrete measurement needs future C atoms and probably A parser setup.
- `B_flow_enable_channels_from_module_inventory`: module inventory discovery is A; pure B can only accept externally validated channels or use station-profile style inputs.
- `B_flow_wgfmu_open_initialize_self_test`: WGFMU session open is A; only initialize/self-test/calibration can be B.
- `B_flow_wgfmu_calibrate_with_error_summary`: WGFMU error/warning summaries are A atoms, so the full bracket belongs in AB.
- `B_flow_easyexpert_abort_wait_fetch_result`: abort is B, but wait/error/result fetch are A.
- `B_flow_easyexpert_open_workspace_zero_cancel`: workspace open/state discovery is A; zero-cancel is B.
- `B_flow_cmu_correct_then_measure_cv`: CMU correction is B, but CV execution/result readout is future C/A composition.

## Missing B_atoms Noted During Planning

- `B_atom_validate_channels_for_state_change`: needed before pure B channel enable/path flows when no A module inventory is allowed; should validate against a station profile or caller-supplied topology.
- `B_atom_query_channel_output_state`: needed for B-only verification of enabled/disabled/zeroed state beyond `B_atom_confirm_zero_outputs`.
- `B_atom_query_auto_calibration_policy`: needed to bracket `B_atom_set_auto_calibration` and restore prior policy without A settings reads.
- `B_atom_restore_auto_calibration_policy`: useful convenience atom if policy bracketing should be atomic/auditable.
- `B_atom_query_smu_filter_state`: needed for reversible SMU filter changes without relying on A current-settings snapshots.
- `B_atom_query_adc_zero_state`: needed for reversible ADC-zero flows.
- `B_atom_validate_path_switch_fixture_state`: needed before ASU/SCUU switching when a human fixture condition or interlock-like station state must be confirmed.
- `B_atom_query_asu_path` and `B_atom_query_scuu_path`: needed to verify path changes and support restoration.
- `B_atom_query_cmu_correction_state`: needed to verify correction enable/disable and restore prior CMU correction policy.
- `B_atom_query_cmu_phase_compensation_mode`: needed to make phase compensation mode changes reversible.
- `B_atom_easyexpert_query_standby_state`: needed to verify and restore EasyEXPERT standby state without A remote status reads.
- `B_atom_easyexpert_query_selected_measurement_state`: needed before/after abort flows to distinguish idle, running, and aborted states.

## Design Notes

- Naming: use action-oriented, endpoint-scoped names such as `B_flow_emergency_abort_and_zero_outputs`, `B_flow_prepare_asu_low_current_path`, and `B_flow_wgfmu_calibration_bracket`. Do not encode numeric ordering in flow names.
- Return schema: each flow should return `flow`, `fake`, `hardware_touched`, `ok`, `partial`, `safety_level`, `inputs`, `outputs`, `cautions`, `operator_acknowledgements`, `atoms_called`, and raw per-atom results in execution order.
- Idempotency: zero/disable and confirmation flows should be safe to rerun. Reset, initialize, calibration, path switching, channel enable, correction clearing, and zero-cancel measurement are stateful and should require explicit inputs or acknowledgements.
- Safety semantics: a B flow may make the instrument or remote application safer, but it must not claim full safety unless the exact B atoms prove that claim. For example, `B_atom_zero_all_outputs` plus `B_atom_confirm_zero_outputs` can support a zero-output claim, but not a disconnected, error-free, or measurement-ready claim.
- AB boundary: if a flow needs transport/session setup, module inventory, status byte, error queue, operation-complete wait, current settings, result buffer, or EasyEXPERT workspace state, keep that composition for later `AB_flow_*`.
- Failure handling: stop on failed safety-critical atoms by default. Return partial results and the first failed atom, but do not auto-recover with reset/calibration unless the flow name and inputs explicitly request that behavior.
- Operator acknowledgements: require explicit acknowledgement fields for destructive or fixture-dependent actions such as reset, self-calibration, CMU correction, phase compensation, QSCV offset cancellation, ASU/SCUU path switching, and channel enabling.
- Parallelism: default to serial execution. Even when two per-channel operations look independent, real B1500A/GPIB/EasyEXPERT/WGFMU sessions should not receive concurrent state-changing commands unless the transport and driver later provide an audited batching layer.
