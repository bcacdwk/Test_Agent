# A Flow Candidates - GPT

## Scope

`A_flow_*` tools should compose only existing `A_atom_*` tools into reusable connection, communication, status, discovery, result-buffer, and remote-session workflows. They may open/close sessions, identify endpoints, collect inventories, read or drain status/error queues, configure parser-facing communication settings such as result format or timestamp output, and read existing output/result buffers.

`A_flow_*` tools must not call `B_atom_*` or future `C_atom_*` tools. They must not reset or initialize the instrument, run calibration or self-test, enable/disable output channels, change SMU/CMU/WGFMU output state, perform safety cleanup, select or execute measurement recipes, or imply that a disconnect is safe without an upstream safety/state-control flow.

Because B1500A query responses use a one-response query buffer, direct B1500A atom calls that issue queries should be serialized on a single transport even when they are logically independent. Parallelism should be limited to client-side parsing or independent sessions/hosts, not concurrent queries to the same instrument session.

## Rejected / Too Broad A_flow Candidates

- `A_flow_safe_shutdown_b1500`: tempting, but a real safe shutdown needs output zero/disable and confirmation atoms from B scope.
- `A_flow_reset_and_discover_b1500`: uses reset/initialize semantics and would cross into B scope even if followed by A discovery.
- `A_flow_run_b1500_self_test_snapshot`: self-test and diagnostics are B atoms, not A status reads.
- `A_flow_measure_then_collect_b1500_buffer`: starting the measurement would require future C measurement atoms; only the buffer collection part belongs in A_flow.
- `A_flow_prepare_measurement_preflight`: readiness gates need interlock, pin-map, limits, and safety policy checks from B scope.
- `A_flow_enable_channels_and_prepare_buffer`: channel enable/disable and output state changes are B scope even though buffer preparation is A scope.
- `A_flow_wgfmu_initialize_and_discover`: WGFMU initialize/self-test/self-calibration are B atoms; only session open, channel discovery, logging, and diagnostics are A scope.
- `A_flow_wgfmu_clear_setup_default`: `A_atom_wgfmu_clear_setup` is an A atom, but bundling it as a default flow is too destructive for unsaved software patterns/sequences and resembles reset-style behavior. Prefer explicit single-atom calls or a later narrowly guarded flow.
- `A_flow_easyexpert_run_application_test`: selecting/running a test is measurement execution, not pure A remote-session discovery/result retrieval.
- `A_flow_easyexpert_abort_and_fetch`: abort is B scope; fetching the latest result alone can be A scope.
- `A_flow_easyexpert_zero_cancel_status_and_measure`: zero-cancel on/off/measure/query belongs to B calibration/state-control scope, not A_flow.

## Missing A_atoms Noted During Planning

- `A_atom_query_service_request_mask`: needed to make `A_flow_configure_b1500_status_polling` reversible and less disruptive to other clients.
- `A_atom_serial_poll_status_byte`: useful for real SRQ interrupt-service flows because `*STB?` and serial poll have different clearing semantics.
- `A_atom_wgfmu_get_warning_level`: needed for a fully non-mutating WGFMU diagnostics snapshot; the structured WGFMU reference includes `WGFMU_getWarningLevel`, but the current MCP surface only exposes set.
- `A_atom_easyexpert_open_remote_session` and `A_atom_easyexpert_close_remote_session`: needed if EasyEXPERT remote should have an explicit socket/session lifecycle like direct B1500A and WGFMU.
- `A_atom_easyexpert_open_preset_group`: likely needed for real `:BENCh:PRESet:SETup:CATalog?` discovery because the remote API model opens a preset group before listing setups.
- `A_atom_easyexpert_fetch_result_siblings`: useful for application-test result discovery/readout beyond only the latest result.

## Design Notes

- Naming: keep flow names action-oriented and scoped by endpoint, for example `A_flow_discover_b1500_session`, `A_flow_snapshot_wgfmu_diagnostics`, and `A_flow_fetch_easyexpert_latest_result`. Do not encode numeric ordering in names.
- Return schema: each flow should return `flow`, `fake`, `hardware_touched`, `inputs`, `outputs`, `warnings`, `caveats`, `parallelizable_steps`, and an ordered `atoms` array containing `{name, inputs, result, status, elapsed_ms}` for auditability.
- Normalized outputs: expose common summary fields at the top level, but preserve every raw atom response so clients can debug parser or fake-surface differences.
- Idempotency: read-only discovery/status flows should be safe to rerun. Flows that clear buffers, drain errors, configure result format, configure SRQ masks, open log files, or open workspaces should require explicit inputs that make the state change obvious.
- Error handling: flows should stop on transport/session errors, but still return all completed atom results. For status/error drain flows, support bounded loops and a truncation indicator once real queue semantics are implemented.
- Parallelism: default to serial atom execution on one B1500A, WGFMU, or EasyEXPERT session. Document any client-side-only parallel work separately from instrument commands.
- Client UX: expose short progress labels such as `connecting`, `discovering_modules`, `reading_status`, `reading_buffer`, `opening_workspace`, and `fetching_result`. The UI should show caveats when a flow is diagnostic-only and does not perform safety cleanup.
- Guardrails: do not let A_flow descriptions imply hardware readiness, DUT safety, or measurement validity. They provide communication/session evidence only.
